# Website And Docs Content Map

> Document 09 | CreditChain Content System | Version 1.0
> Scope: main website IA, docs portal IA, canonical product positioning, reusable copy

---

## 1. Canonical Product Model

Every public-facing CreditChain property should present the platform as four
connected surfaces:

1. **CreditChain Core**: the Layer 1 protocol
2. **Creditscan**: the production blockchain browser
3. **creditchain-ts-sdk**: the developer SDK
4. **docs.creditchain.org**: the documentation system

The website and docs portal should not treat these as unrelated repos.

## 2. Canonical Headline Options

Approved headline directions:

- "The public chain for stablecoins, settlement, and programmable finance."
- "Institution-grade Layer 1 infrastructure for stablecoins, digital assets, and AI-era banking."
- "Public chain, explorer, SDK, and docs system for CreditChain financial infrastructure."

Avoid vague wording such as "next generation chain" without specifying what it
is for.

## 8. Naming Rules

Use these exact spellings:

- `CreditChain`
- `Creditscan`
- `creditchain-ts-sdk`
- `CCC`
- `docs.creditchain.org`
- `creditchain.org/docs`

Avoid mixing in older brand names in public-facing copy.

## 9. Source Of Truth

When content conflicts exist:

1. protocol and product facts come from this repository and the related local
   Creditscan / SDK projects
2. website summaries should follow this document
3. per-component technical details belong in each product guide or component README
